# 期货分钟线 ClickHouse 存储大表与数据索引设计

为了支持多品种、海量期货分钟线（1-minute bar）的统合存储与高效检索，我们在此记录关于数据库架构设计、网络寻址、大规模导入并发控制以及元数据索引的相关技术决策。

---

## 1. 物理表结构与分区设计
我们决定创建统一的物理大表 `default.futures_1m_bar`，承载全市场所有期货品种的历史及实时分钟线时序数据：
* **引擎选择**：使用 `MergeTree` 引擎。
* **物理主键与排序**：采用 `ORDER BY (code, datetime)` 联合物理主键。在物理存储上先按合约分块，再按时间线性排列。这保证了量化回测系统执行“单合约历史切片查询”时能够获得毫秒级的极速 I/O 检索。
* **分区设计**：采用 `PARTITION BY toYear(datetime)` 按年分区。
  * *权衡背景*：由于单品种包含数百个到期合约，若按月分区会导致物理小文件和目录数发生灾难性膨胀，严重拖慢合并进程。按年分区不仅能将分区数量控制在十个以内，又兼顾了冷热备份、物理 Drop 历史分区的便利性。

---

## 2. 交易所代码官方映射与数据纯净度
在量化研究中，数据的物源可追溯性（Provenance）是生命线。由于聚宽等原始数据源常使用非标的交易所缩写（如 `XZCE`、`XSGE`、`CCFX` 等），我们在 `schema.py` 中实现了标准代码解析与转换机制：
* **规则映射**：通过 `EXCHANGE_SUFFIX_MAP` 全自动且显式将后缀替换为交易所官方标准后缀（例如 `.XZCE` 转换为 `.ZCE`，`.CCFX` 转换为 `.CFX`）。
* **物源可控性**：所有转换操作在 Polars DataFrame 级别中由统一的 `normalize_code_expr("code")` 表达式执行，保证了数据在进入 ClickHouse 之前已完成清洗，数据库中不存在非标脏数据。

## 3. 大数据量导入并发控制与异常防御

### 3.1 批量一次性清理 (ALTER TABLE DELETE IN)
* *死锁缺陷*：最初在写入合约 CSV 前逐个执行 `DELETE WHERE code = 'xxx'`，遇到国债期货等单包 400+ 合约的品种时，会瞬间触发 ClickHouse 突变（Mutations）队列爆满，导致客户端 Socket 挂起死锁。
* *重构决策*：改为在处理单个 ZIP 前预扫描提取所有 `clean_codes`，合并为一条 `ALTER TABLE ... DELETE WHERE code IN ('c1', 'c2', ...)` 一次性执行批量清理，将突变执行频率降低了 100~400 倍，并显式追加 `time.sleep(1.0)` 等待后台 Mutations 调度就绪。

### 3.2 物理切片与指数退避重试 (safe_insert_df)
* *网关超时（524）与连接重置（502）*：在公网代理（如 Cloudflare 保护层）环境下，单次向 ClickHouse 插入几百万行主力/指数合约数据会触发 100 秒的 HTTP 连接超时上限；且瞬间并发量大会导致服务器主动 reset 端口返回 502。
* *重构决策*：封装了具有高容错性的 `safe_insert_df()` 函数：
  1. **自动切片**：若 DataFrame 行数超过 10 万行，自动执行 Polars 物理切片，拆分为 10 万行一个小批次极速发送（单次仅需 1~3 秒）。
  2. **指数加倍退避**：对于任何网络抖动或临时拒绝，捕获异常并自动以指数延时（2s -> 4s -> 8s -> 16s）重试最多 5 次，并在每次成功写入后微休眠 50ms 释放连接池压力。

### 3.3 孤儿品种提取 (IF 股指期货)
* *打包遗留*：中金所股指期货 `IF` 并没有自己专属的 ZIP 压缩包，而是被混塞在国债 `TF`/`TL`/`T` 包中。
* *重构决策*：利用正则对 CSV 文件名进行严格前缀过滤，避免混入无关数据。特别地，仅在处理 `TF` 品种时将匹配正则扩大为 `^(TF|IF)` 顺带合并导入 `IF`（在 `TL`、`T` 包中过滤 `IF` 避免重复）。这既保证了 `IF` 股指期货数据的完整性，又保持了单次写入的幂等性。

## 4. 自适应双通道网络寻址
为了解决多机房、内外网部署切换繁琐的问题，脚本在握手 ClickHouse 时实现了自适应降级寻址：
* **第一优先级（内网极速）**：优先碰本地局域网 `192.168.100.99:8123` 并强制设定 `connect_timeout=3`（3 秒超时，防止在外网环境下挂死）。
* **第二优先级（公网备用）**：内网超时或报错后，捕获异常，自动降级至配置文件（[configs/futures_smoke.yaml](file:///Users/link/Documents/FuturesModelFactory/configs/futures_smoke.yaml)）中配置的远程公网域名（如 `ch.jieyujing.eu.org:443`）进行连接。

## 5. 独立元数据索引表
为了解决由于主表行数庞大（千万至数亿级别）导致 GROUP BY 合约起止时间和行数耗时严重的问题，我们设计并建立了独立的元数据索引表 `default.futures_1m_bar_index`：
* **引擎**：采用 `ReplacingMergeTree` 引擎并按 `code` 排序。
* **数据刷新**：在每次全量或增量导入的最后阶段，由 Python 自动触发 `TRUNCATE TABLE` 后以 `INSERT INTO ... SELECT code, min(datetime), max(datetime), count() GROUP BY code` 从主表完成聚合刷新。
* **效益**：将量化系统对“有哪些合约与对应起止时间”的查询耗时，从数秒至十几秒降低到 **1 毫秒级**，极大地提升了回测引擎启动时的加载效率。
